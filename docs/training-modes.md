# Training modes

AsyncOPD exposes three public training families: OPD, GRPO-style reward
training, and SFT. DAPO-style runs use the GRPO mode with additional algorithm
settings. Each mode is selected by `algorithm.mode` and routed through a
coordinator mode, trainer, backend, loss, and batch-preparation path.

## Mode selection

| Goal | Use | Public starter |
| --- | --- | --- |
| Distill a larger teacher into a smaller student using on-policy samples | OPD | `configs/examples/opd_qwen3_1.7b.yaml` |
| Improve reasoning/math behavior from rewards | GRPO | `configs/examples/grpo_gsm8k_0.5b_4gpu.yaml` or `configs/examples/grpo_qwen3_1.7b.yaml` |
| Run DAPO-style math RL on the local DeepMath subset | DAPO | `configs/examples/dapo_deepmath_qwen3_4b_8gpu.yaml` |
| Fine-tune on known completions | SFT | `configs/examples/sft_qwen3_1.7b.yaml` |

## OPD: on-policy distillation

OPD trains a student on responses generated by the current student policy. A teacher model scores those responses, and the trainer applies a KL-style objective.

### Flow

1. Rollout worker generates responses with the student model.
2. Teacher worker computes reference logprobs for the generated responses.
3. Coordinator assembles aligned train batches.
4. Trainer computes OPD loss and updates student weights.
5. Weight sync broadcasts fresh student weights to rollout workers.

### Starter command

```bash
python -m opd.cli.train --config configs/examples/opd_qwen3_1.7b.yaml --overwrite
```

For a short smoke on the same config:

```bash
python -m opd.cli.train --config configs/examples/opd_qwen3_1.7b.yaml --overwrite \
  --set trainer.total_steps=1 trainer.total_epochs=1 eval.freq=-1 eval.before_train=false trainer.save_freq=-1
```

For multi-sample PG-KL or advanced scheduling, start from
`configs/examples/opd_deepmath_qwen3_1.7b_pg_mc64_stepoff_8gpu.yaml` or
`configs/examples/opd_deepmath_qwen3_1.7b_pg_mc64_async_8gpu.yaml`.

### Important OPD settings

```yaml
algorithm:
  mode: opd
  opd:
    kl_loss_mode: forward_kl
```

| Setting | Why it matters |
| --- | --- |
| `algorithm.opd.kl_loss_mode` | Selects the distillation objective. Start with `forward_kl`. |
| `teacher.vllm.n_logprobs` | Controls teacher support size for logprob-based losses. |
| `data.max_prompt_length` / `data.max_response_length` | Must fit teacher and rollout context budgets. |
| `trainer.use_sequence_packing` | Improves efficiency but makes alignment tests more important. |
| `trainer.kl_chunk_size` | Chunks LM-head/log-softmax loss work over sequence positions to reduce trainer memory. See [Loss/logit chunking](loss-chunking.md). |
| `pipeline.n_step_off.step_off` | Controls overlap/staleness. Start small. |
| `weight_sync.backend` | `nccl` is the performance path. |

### OPD loss modes

| Loss mode | Use case |
| --- | --- |
| `forward_kl` | Default teacher-covering objective. |
| `reverse_kl` | More mode-seeking behavior. |
| `policy_gradient_kl` | PPO-style policy-gradient formulation using teacher signal; see AReaL/M2PO advanced examples for common variants. |
| `multi_sample_policy_gradient_kl` | Multi-sample PG-KL; see the DeepMath step-off/fully-async examples. |
| `multi_sample_forward_kl` | Multi-sample forward KL; see `configs/examples/advanced/opd_deepmath_qwen3_4b_multisample_forward_kl_8gpu.yaml`. |
| `reverse_kl_rollout_student_topk` | Sparse/top-k reverse KL; see `configs/examples/advanced/opd_deepmath_qwen3_4b_reverse_kl_topk_8gpu.yaml`. |
| `thunlp_opd_default_loss` | THUNLP-style top-k PPO OPD loss; see `configs/examples/advanced/opd_deepmath_qwen3_4b_thunlp_default_8gpu.yaml`. |

### Advanced OPD examples

Advanced OPD examples live under `configs/examples/advanced/`. They are for specific research features such as AReaL-style decoupling, M2PO clipping, sparse/top-k KL, dense hidden recompute, code training, and fused hybrid sync. Start with a starter config first, then switch to the advanced config that matches the feature you want to test.

### OPD debugging checklist

- Set `pipeline.n_step_off.step_off: 0` for first correctness debugging.
- Reduce `trainer.total_steps`, `eval.freq`, `rollout.vllm.max_num_seqs`, and context lengths for launch smoke.
- Lower `trainer.kl_chunk_size` if OPD trainer OOM happens in the KL/logprob head rather than the transformer body.
- Inspect response masks and teacher masks if loss values look wrong.
- Keep tokenizer padding left-aligned.
- Confirm teacher and rollout model/tokenizer choices are compatible.

## GRPO and DAPO-style training

GRPO samples multiple responses per prompt, scores them with a reward function, computes group-relative advantages, and trains with PPO-style clipping. DAPO-style behavior is represented through GRPO settings such as asymmetric clipping, token-mean aggregation, and group filtering.

### Flow

1. Rollout generates `group_size` responses per prompt.
2. Reward function scores each response.
3. Advantages are normalized within each prompt group.
4. Trainer applies PPO/GRPO loss with clipping and optional KL/reference penalty.
5. Weight sync updates rollout workers.

### Starter commands

Smallest public GPU smoke target:

```bash
python -m opd.cli.train --config configs/examples/grpo_gsm8k_0.5b_4gpu.yaml --overwrite \
  --set trainer.total_steps=1 trainer.total_epochs=1 eval.freq=-1 eval.before_train=false trainer.save_freq=-1
```

Larger Qwen3 example:

```bash
python -m opd.cli.train --config configs/examples/grpo_qwen3_1.7b.yaml --overwrite
```

DAPO-style DeepMath example, after building
`data/deepmath_difficulty6/train.parquet`:

```bash
python -m opd.cli.train --config configs/examples/dapo_deepmath_qwen3_4b_8gpu.yaml --overwrite
```

### Important GRPO settings

```yaml
algorithm:
  mode: grpo
  grpo:
    group_size: 5
    clip_eps: 0.2
    kl_beta: 0.001
    kl_type: low_var_kl
    reward_fn: correctness
```

| Setting | Why it matters |
| --- | --- |
| `group_size` | Number of samples per prompt. Larger values increase reward signal and generation cost. |
| `reward_fn` | Reward implementation. Public examples use `correctness`. |
| `answer_pattern` | Regex override for extracting answers such as `#### <number>`. |
| `clip_eps` | PPO-style clipping. |
| `clip_ratio_low` / `clip_ratio_high` / `clip_ratio_c` | DAPO-style asymmetric/dual clipping knobs. |
| `kl_beta` | Reference KL strength. If `0.0`, teacher/reference can be omitted. |
| `filter_groups` | DAPO-style group filtering. |
| `loss_agg_mode` | Token/sample aggregation behavior. |
| `norm_adv_by_std` | Advantage normalization within groups. |

### Reward behavior

The default `correctness` reward uses answer extraction and matching suitable for math datasets:

1. `\boxed{...}` extraction;
2. `#### <answer>` extraction;
3. last-number fallback;
4. numeric/symbolic matching helpers where available.

When adding a new reward, keep it deterministic, batch-friendly, and easy to
test. Handle empty generations and malformed answers, support the dataset answer
format through `answer_key` or `answer_pattern`, and add tests for exact match,
formatting variants, invalid output, batch behavior, and metric aggregation.

### Running without a teacher/reference

For GRPO configurations that do not need reference KL:

```yaml
algorithm:
  mode: grpo
  grpo:
    kl_beta: 0.0
```

This can reduce GPU requirements, but changes the optimization objective. Keep this explicit in experiment notes.

## SFT: supervised fine-tuning

SFT trains directly on prompt/completion pairs. It does not start rollout or teacher workers.

### Starter command

```bash
python -m opd.cli.train --config configs/examples/sft_qwen3_1.7b.yaml --overwrite
```

The public SFT config is a template. Before running, replace:

```yaml
data:
  train_files: data/train.parquet
  val_files: data/val.parquet
  prompt_key: prompt
  completion_key: completion
```

with paths and columns from data you control.

### SFT loss modes

```yaml
algorithm:
  mode: sft
  sft:
    loss_mode: ce
```

| Loss mode | Description | Public guidance |
| --- | --- | --- |
| `ce` | Standard completion cross-entropy | Recommended starting point. |
| `kl` | Off-policy KL from stored teacher logits | Trusted-data workflow only. |
| `mixed` | Blend of CE and KL terms | Advanced; test carefully. |

### Teacher-solution/off-policy SFT data

For off-policy distillation, generate prompt/completion parquet files from a
teacher model:

```bash
python scripts/gen_teacher_solutions.py \
  --teacher Qwen/Qwen3-30B-A3B-Instruct-2507 \
  --data data/deepmath_difficulty6/train.parquet \
  --output-dir data/sft_teacher_solutions \
  --n 20000 \
  --n-val 500 \
  --gpus 0,1,2,3 \
  --dp 4
```

The output contains `train.parquet` and `val.parquet` with `prompt`,
`completion`, and `answer` columns. Point the SFT template at those files:

```yaml
data:
  train_files: data/sft_teacher_solutions/train.parquet
  val_files: data/sft_teacher_solutions/val.parquet
  prompt_key: prompt
  completion_key: completion
```

`scripts/gen_teacher_solutions.py --save-logits` also writes pickle-serialized
teacher top-k tensors for legacy SFT-KL experiments. Treat those parquet files
as trusted-only artifacts and keep `data.allow_pickle_teacher_logits: false`
unless you intentionally generated and control the data.

## Evaluation during training

Common eval settings:

```yaml
eval:
  freq: 40
  mode: [post_allgpu]
  before_train: true
  n_samples: 32
  temperature: 1.0
```

| Mode | Use |
| --- | --- |
| `inline` | In-run validation while workers are active. |
| `post` | Post-training evaluation from saved checkpoints. |
| `post_allgpu` | Shut down training workers, then use all GPUs for faster eval. |
| `perplexity` | SFT-style validation. |

For launch debugging, temporarily disable eval:

```text
--set eval.freq=-1 eval.before_train=false
```

## Choosing hardware and batch sizes

Start from the public example nearest your hardware and edit downward:

- Lower `rollout.vllm.max_num_seqs` for vLLM OOM.
- Lower `data.max_response_length` or `rollout.vllm.max_model_len` for long-context OOM.
- Lower `trainer.micro_batch_size` for trainer OOM.
- Lower `trainer.kl_chunk_size` for trainer OOM tied to KL/logprob computation; this chunks the LM head over sequence positions.
- Lower `algorithm.grpo.group_size` for GRPO generation pressure.
- Keep trainer/rollout GPU IDs disjoint unless a config intentionally colocates them.

## Reading run outputs

| File | What to inspect |
| --- | --- |
| `run.log` | Process startup, model loading, warnings, stack traces. |
| `log.jsonl` | Training metrics, eval metrics, step progress. |
| `trace.json` | Timing breakdown and async overlap in Perfetto/Chrome trace viewers. |
| `validation_outputs/` | Per-sample eval generations. |
| `checkpoints/` | Saved model/optimizer state when checkpointing is enabled. |
